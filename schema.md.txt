## What "MVP" means for CurateKin

The one loop that has to work for CurateKin to be a real product:

**Creator curates a product → publishes it on their storefront → a shopper finds and browses it → shopper clicks → shopper reaches the real product → (eventually) creator earns commission.**

Everything else — follower counts, earnings dashboards, chat, gifting — is decoration until this loop works end to end. As of today, **this loop works**, with one piece (real commission tracking) still pending an external approval.

## What got done this session — email confirmation, fully fixed

1. **Found the real cause of confirmation links failing:** both app/signup/page.tsx and app/signup/creator/page.tsx build the confirmation redirect address dynamically from window.location.origin at the moment of signup — nothing hardcoded, no environment variable.
2. **Discovered that curatekin.com (no www) forwards to https://www.curatekin.com** before the app's own code ever runs — meaning on the live site, the address actually sent to Supabase is always https://www.curatekin.com/signup/confirm, never the bare non-www version.
3. **Checked Supabase's Redirect URLs allow-list directly and found it completely empty** — an earlier attempt to add an entry had never actually saved.
4. **Added https://www.curatekin.com/signup/confirm to the Redirect URLs list** and saved it.
5. **First retest showed an "invalid or expired link" error** — traced to clicking an old confirmation email from an earlier test sitting in the inbox, not a new bug.
6. **Retested clean with a brand-new signup:** confirmation link now correctly lands on the live curatekin.com site and shows "Email confirmed." Verified via screenshot. This test went through the creator signup path specifically.
7. **Not yet separately verified:** the shopper signup path uses the same code pattern and should behave identically, but hasn't been tested directly — added as an open item below.

## Honest assessment — risks worth naming

**Things get lost between sessions.** Today we discovered a fully-built public storefront page that already existed and worked, and an access-key gate (`SITE_LIVE = false` in `proxy.ts`, key: `billiondollarcompany`) quietly blocking every real visitor — neither documented anywhere before today. This is a real pattern, not a one-off: undocumented work gets "re-discovered" instead of built on.

**Today's work wasn't linear.** We moved from avatar styling → SQL diagnostics → auto-fill → category logic → storefront discovery → a critical redirect bug, mostly reactively as each bug surfaced the next. Partly unavoidable, partly avoidable with more upfront planning per session.

**The terminal/git/VS Code layer has been a consistent source of friction** — "why won't localhost load," "what does git push do," files pasted into the wrong folder. This is a normal place to be without a dev background, but it has cost real time today and is worth solving directly rather than repeatedly working around.

**Fix going forward:**
- Start each session by opening this doc and picking one thing to focus on, rather than letting scope drift
- Keep "check the actual file/database before writing new code" as the default for everything, not just schema questions
- Seriously consider having one technical person handle git/deployment/environment setup for a few hours, freeing time for product decisions instead of terminal commands

## Supabase-specific friction (raised today)

Supabase itself is a reasonable choice and isn't the core problem — but today's *use* of it was slow, for identifiable reasons:

- **No schema map existed until partway through today.** Every bug required stopping to dig through SQL Editor to find real column names (`full_name` vs `display_name`, missing `followers` column, etc.) — reactive discovery instead of known facts. `SCHEMA.md` should fix this, but only if kept updated.
- **Supabase recently split API keys into two systems** — new "Publishable/Secret" keys vs the old "Legacy anon/service_role" keys — and the codebase expects the old naming. This is a genuinely confusing transition Supabase itself made, not a mistake on our end.
- **Raw SQL queries were used for things the visual Table Editor would answer in seconds** (e.g. "what columns does this table have"). SQL is right for actual changes; it's overkill for a quick look.

**Fix going forward:**
- Use the **Table Editor** tab (not SQL Editor) for quick "what's in this table" checks
- Keep `SCHEMA.md` updated any time a table changes, so we stop re-diagnosing from scratch
- Write SQL only when actually changing the database, always explained clearly before running

## Process notes (things to keep doing / avoid)

- **Check the actual file/database before writing new code.** Several bugs today (column names, the redirect table, the access key) were found by reading existing files first instead of guessing — this should stay the default move, not the exception.
- **Test locally when possible**, using `npm run dev` and `localhost:3000`, before pushing to the live site.
- **Remember `git add . && git commit -m "..." && git push`** is the real save-and-deploy sequence — "git save" is not a real command.
- **The schema reference file** (`SCHEMA.md`, saved in the project root) should get updated if the database structure changes again.
